제품의 본질을 이해하고 비즈니스 임팩트를 고려한 '소통 가능한 코드'를 지향하는 서버 개발자입니다. 안정적이고 확장 가능한 백엔드 시스템 구축에 열정을 가지고 있으며, 특히 대용량 데이터 처리와 배치 시스템 최적화에 강점이 있습니다. 최근에는 27년된 레거시 SQL Server Stored Procedure를 Spring Batch 구조로 리빌딩했습니다.

리빌딩한 정산시스템은 Jenkins와 GitHub Actions로 배치 스케줄링을 자동화하고, 각 Job·Step의 처리량·오류율·재시도 건수·소요시간 등의 메트릭을 수집합니다.
수집된 메트릭은 Telegram으로 실시간 알람을 전송하여 운영 이슈를 빠르게 인지할 수 있도록 했습니다.
또한 Metabase 기반 대시보드를 통해 일별 처리 현황과 정합성 지표를 시각화하여 장애 진단과 데이터 품질관리를 효율화했습니다.

  • Batch Throughput 튜닝

    Partitioner 기반 병렬 파이프라인과 I/O 바운드가 높은 작업을 분리하여 월 3천만 건 데이터를 무중단 집계했습니다.

  • 모듈러 모놀리스(Modular Monolith) 아키텍처 설계 및 개발

    Core/Domain/Batch 멀티모듈로 책임을 정리하고, CI 파이프인, 정적 분석(Sonarqube), 테스트 코드를 연동해 회귀 리스크를 최소화했습니다.

  • Observability 향상 & QA 자동화

    반복적인 정산 검증 작업을 템플릿화해 배치 운영 효율을 높였고, 로그·알람·메트릭을 통합한 대시보드로 장애를 실시간 감지하여, 문제 대응 시간을 시간 단위 → 분 단위로 단축했습니다.

(주)다날 | 플랫폼개발팀 백엔드 개발자

2024.01 ~ 현재

27년간 SQL Server 기반으로 운영되던 정산시스템을 Spring Batch 기반 자동화 플랫폼으로 재구축했습니다. 결제 데이터의 집계, 검증, 지급, 환불까지 모든 정산 흐름을 데이터 중심 구조로 전환했습니다.

<거래내역 생성 구조 개선 및 병렬 처리 아키텍처 설계>

기존 거래내역 생성은 프로시저 기반으로 하나의 테이블을 여러개의 View로 세분화하는 과정에서 데이터 품질 이슈와 병목이 발생했습니다. 이를 “병렬화 불가능한 구조적 한계”로 정의하고 단일 테이블 기반으로 스키마를 설계하고, Spring Batch 기반 병렬 파이프라인으로 재구성했습니다.

Partitioner + Chunk 구조를 도입해 생성–검증–저장을 분리하고, 키 단위 파티셔닝으로 데이터 중복 없이 병렬 실행되도록 설계했습니다. 또한 실패 청크만 재처리할 수 있는 복구 로직을 추가했습니다.

→ 수백만 데이터 기준처리속도 50% 이상 향상, 복구시간 수분 → 수초 단축, 프로세스 추적 가능 구조 확보 (수치개선)

[기술] Spring Boot 3.2, Spring Batch, JPA(QueryDSL), Oracle 19c, PostgreSQL(EPAS)

<외부 데이터 검증(대사) 파이프라인 구축>

카드사, 통신사등 외부시스템(원천사)과의 데이터 불일치가 실시간으로 감지되지 않았습니다. 이를 “정합성 검증 부재로 인한 리스크”로 정의하고, 외부 데이터를 비교·검증하는 파이프라인을 설계했습니다.

NoOffset 쿼리와 Clustered Index를 적용해 속도를 높이고, 누락건 감지 시 Telegram알림으로 자동 통보되도록 했습니다. 비교는 단순 파일대조가 아닌 Key 기반 매칭 알고리즘으로 구현했습니다.

→ 집계시간 170초 → 4초(97% 단축), 실시간 검증·알림 체계 완성

[기술] Spring Batch, QueryDSL, Oracle 19c, PostgreSQL(EPAS), Telegram, Naverworks API

<지급 데이터 보정로직 구현>

기존 지급·환불 로직은 정상 거래건과 1대1 매칭되지 않아 중복 데이터 발생 시 전체 Job이 중단되었습니다. 이를 “건 단위 복구 불가능 구조”로 정의하고, SkipPolicy와 RetryPolicy를 적용해 건 단위 오류 복구 구조를 구현했습니다. (애매하다)

데이터 필수값 검증을 위한 InvalidItemFilterProcessor를 추가하고, 누락·오류 데이터를 자동 보정했습니다.

→ 운영 중단 없는 무중단 지급 배치 구조 확보

[기술] Spring Batch, JPA, Custom SkipPolicy, Logging/Alarm System

<마감 집계 최적화 및 일관성 자동화>

마감 집계는 대량 데이터를 단일 트랜잭션으로 처리하여 장애 시 전체 롤백되는 구조였습니다. 청크 크기와 커밋 간격을 최적화하고, 집계와 상태 업데이트를 분리해 부분 재처리가 가능하도록 했습니다.

→ 집계속도 30% 향상, 누락률 0%, 완전 자동화형 마감 프로세스 구축

[기술] Spring Batch, Oracle 19c, JDBC Template

<가맹점 파일 제공 시스템 리팩토링>

기존 SCP 수동 전송 방식을 RESTful API 기반 자동화 구조로 전환했습니다. 파일 생성–검증–전송–삭제를 일관된 파이프라인으로 구성하고, Caffeine Cache로 반복 요청 부하를 최소화했습니다.

→ 재전송 대응시간 3분 → 10초, 운영 효율 대폭 향상

[기술] Spring Boot 3, REST API, Spring Batch, Caffeine Cache, Oracle 19c

<테스트 환경 및 멀티모듈 아키텍처 도입>

단일 모듈 구조에서 Core/Domain/Batch로 분리해 책임을 명확히 했습니다. 테스트 전용 H2·PostgreSQL 환경을 구성하고, EntityFixtureGenerator로 복잡한 데이터를 자동 생성하여 테스트 코드 작성시 생산성도 높였습니다.

→ 빌드시간 25% 단축, 테스트 작성 공수 대폭 감소

[기술] Spring Boot, Gradle Multi-module, H2, PostgreSQL, Spring Batch Test

<모니터링 및 알림 시스템 구축>

배치 오류 추적이 어려웠던 구조를 개선하기 위해 각 StepExecution 이벤트를 수집하여 Telegram 알림으로 통합하고, 정산별 Job 상태와 실패건수를 실시간 모니터링했습니다. 이를 통해 현재까지 700여건이 넘는 실패를 파악하고 복구작업을 수행할 수 있었습니다. 복구작업의 자동화 Metabase를 도입해 검증 쿼리 70여 개를 시각화했습니다.

→ 장애 대응시간 수십분 → 1분, 데이터 검증 가시성 확보

[기술] Telegram Bot API, Metabase, Spring Batch Listener, JPA

(주)쿠콘 | 마이데이터연구부 백엔드 개발자

2021.08 ~ 2022.10

삼성카드 마이데이터 플러그인 웹 서비스(월간 활성 사용자 약 5만 명)를 런칭하고 운영했습니다. 은행·통신·보험·캐피탈 등 이기종 데이터 소스를 통합 수집/정규화하는 백엔드 API를 개발했고, 민감 금융데이터에 대해 암호화·정합성·FTP 전송 프로세스까지 엔드투엔드로 책임졌습니다.

<업권별(은행·통신·보험·캐피탈) 마이데이터 수집 API 설계 및 정규화>

금융감독원에서 표준화된 API를 제공함에도, 각 정보제공자의 업권마다 값의 정의나 포맷이 달라 공통 로직이 누적될수록 유지보수 난이도가 증가했습니다.

이를 해결하기 위해 업권별 수집 로직을 전략(Strategy) 패턴 기반 모듈로 분리하고, 필수 스키마(거래일시, 금액, 기관 식별자 등)를 표준화한 공통 DTO로 매핑하는 파이프라인을 설계했습니다.

또한 사용자의 실제 시나리오(잔액 조회, 카드 승인/취소 내역 조회 등)에 맞춰 오류 케이스를 세분화하고 명확한 응답 메시지를 정의해 고객센터/QA의 재현 가능성을 높였습니다.

→ 신규 기관 연동 시 공통 코드 재사용이 가능해져 온보딩 속도와 유지보수성이 향상되었고, 사용자 관점에서의 에러 메시지 일관성 확보로 CS 대응 난이도도 낮아졌습니다.

[기술] Spring Boot 2.6, Java 8, PostgreSQL, JUnit5

<거래·계좌 데이터 무결성/정합성 검증 및 대시보드 구축>

초반에는 외부 제휴사에서 내려준 원장성 데이터(거래내역, 잔액 등)가 간헐적으로 누락·중복·형식 오류를 포함하고 있어 운영 단계에서만 발견되는 문제가 있었고, 이는 금융 서비스 신뢰성과 직결되는 리스크였습니다.

이를 “수집 이후에나 발견되는 정합성 리스크”로 정의하고, 수집 직후 무결성을 점검하는 검증 로직과 이를 가시화하는 내부 대시보드를 개발했습니다

- 거래내역성 데이터 중심으로 테이블 스키마를 재설계해 “원거래 ↔ 수집데이터” 간 매핑이 추적 가능하도록 개선
- 각 검증 포인트마다 상세 로깅 + Unchecked Exception 기반 예외 트래킹을 추가해 오류 상황을 자동 분류/라벨링
- 운영팀이 실시간으로 상태를 확인할 수 있도록 무결성 대시보드 페이지를 구성

→ 반복적으로 발생하던 정합성 이슈를 사전에 감지 가능해졌고, QA 단계에서 발생하던 수기 확인 업무가 줄어 QA 오류 지적 건수를 약 90% 이상 감소시켰습니다.

[기술] Spring Boot 2.6, PostgreSQL, JUnit5, Monitoring Dashboard(Javascript, jQuery)

<민감데이터 암호화 및 배치 전송 자동화 (AES-256)>

마이데이터 서비스 특성상 개인 신용·자산 정보가 삼성카드에 양방향 암호와되어 전달되기 때문에, 개인정보 보호 규제 준수와 안전한 전달(메일링/배치)이 핵심 과제였습니다.

이를 위해 AES-256 / SHA-256 기반의 암호화 모듈을 직접 설계·적용하고, 제휴사 제공용 배치 프로세스를 만들었습니다. 이 프로세스는 민감 데이터를 암호화 → 전송용 산출물 생성 → 메일링 → 이력 저장까지 자동화된 파이프라인으로 동작하게 처리하였습니다.

또, CBT(사전 검증 환경) 테스트, 이메일 인증, 보안 키패드 등 다단계 인증 플로우를 서비스에 적용해 사용자 접근제어를 강화했습니다.

→ 개인정보 전달/보관 과정에서의 인적 실수 여지를 제거했고, 보안·준법 감사를 위한 근거(로그/추적성)를 체계화하여 운영 부담을 크게 줄였습니다.

[기술] Spring Boot 2.6, AES-256/SHA-256, 배치 파이프라인 설계, 메일 발송 자동화

<WebView 기반 고객 화면 및 환경설정 UI 개발>

마이데이터 서비스는 카드사 앱/WebView 안에서 구동되는 구조였기 때문에, 단순한 API 제공을 넘어 실제로 노출되는 조회 화면/환경설정 화면까지 만들어야 했습니다.

업권별 데이터 특성(예: 통신 요금제/납부내역, 금융 거래내역 등)에 맞춘 UI와 사용자 설정 페이지를 개발하고, 반복적인 사용자/내부 QA 사이클을 통해 민감한 정보 조회 흐름의 사용성을 개선했습니다.

→ 카드사 고객이 직접 쓰는 실서비스 품질까지 관여하며, 백엔드뿐 아니라 사용자 경험(UX) 단계까지 책임지는 풀스택 역량을 확보했습니다.

[기술] Spring Boot, jQuery, WebView 통합

<운영 프로세스/배포 체계 정립 및 문서화>

마이데이터는 금융 규제, 제휴사 요구사항, 내부 QA, 운영팀 요청이 얽혀 있어 “누가/언제/어떤 근거로 수정했는가”를 명확히 남기는 게 매우 중요했습니다.

그래서 이슈 티켓 관리 기준, 정기 배포 절차, 장애 시 대응 플로우를 문서화해 팀 내 공통 레퍼런스로 만들고 공유했습니다.

→ 배포 안정성이 올라갔고, 신규 인원이 빠르게 온보딩할 수 있는 팀 운영 가이드가 마련되었습니다.

[총괄 기술] Spring Boot 2.6, Java 8, Oracle, JUnit5, AES-256/SHA-256, WebView/Thymeleaf
