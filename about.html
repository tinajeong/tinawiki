결제 정산 도메인에서 배치/API를 설계·운영하는 백엔드 엔지니어입니다. 레거시 SQL Server 기반 Stored Procedure를 Spring Batch 구조로 이관하며, 배치 오케스트레이션과 데이터 품질 제어, 가시화 체계를 엔드투엔드로 구축해 왔습니다.

지표는 처리량과 정확도으로 정의하고, 비즈니스 KPI와 연동해 리팩토링 우선순위를 결정합니다. 배치 Job·Step 메트릭을 표준화해 장애 진단과 용량 계획을 수치 기반으로 실행합니다.

  • Batch Throughput 튜닝

  • 모듈러 아키텍처 설계

    Core/Domain/Batch 멀티모듈로 책임을 정리하고, CI 파이프라인·정적 분석 규칙을 연동해 회귀 리스크를 최소화했습니다.

  • Observability & QA 자동화

    검증 배치를 템플릿화하고 메트릭·로그·알림을 통합 대시보드로 연결하여 장애 대응 MTTR을 분 단위로 단축했습니다.

(주)다날 | Smart Settlement팀 백엔드 개발자

2023.06 ~ 현재

27년간 SQL Server 기반으로 운영되던 정산시스템을 Spring Batch 기반 자동화 플랫폼으로 재구축했습니다. 결제 데이터의 집계, 검증, 지급, 환불까지 모든 정산 흐름을 데이터 중심 구조로 전환했습니다.

<거래내역 생성 구조 개선 및 병렬 처리 아키텍처 설계>

기존 거래내역 생성은 단일 프로시저 기반으로 병목이 발생했습니다. 이를 “병렬화 불가능한 구조적 한계”로 정의하고 Spring Batch 기반 병렬 파이프라인으로 재설계했습니다.

Partitioner + Chunk 구조를 도입해 생성–검증–저장을 분리하고, 키 단위 파티셔닝(merchant_id 기준)으로 데이터 중복 없이 병렬 실행되도록 설계했습니다. 또한 실패 청크만 재처리할 수 있는 복구 로직을 추가했습니다.

→ 처리속도 40% 향상, 복구시간 수분 → 수초 단축, 프로세스 추적 가능 구조 확보

[기술] Spring Boot 3.2, Spring Batch, JPA(QueryDSL), Oracle 19c, PostgreSQL(EPAS)

<외부 데이터 검증(대사) 파이프라인 구축>

외부 PG 및 연동시스템과의 데이터 불일치가 실시간으로 감지되지 않았습니다. 이를 “정합성 검증 부재로 인한 리스크”로 정의하고, 외부 데이터를 비교·검증하는 파이프라인을 설계했습니다.

NoOffset 쿼리와 Clustered Index를 적용해 속도를 높이고, 누락건 감지 시 Slack/TG 알림으로 자동 통보되도록 했습니다. 비교는 단순 파일대조가 아닌 Key 기반 매칭 알고리즘으로 구현했습니다.

→ 집계시간 170초 → 4초(97% 단축), 실시간 검증·알림 체계 완성

[기술] Spring Batch, QueryDSL, Oracle 19c, PostgreSQL(EPAS), Slack/TG API

<지급/역환 트랜잭션 복구 로직 개선>

기존 지급·환불 로직은 중복 데이터 발생 시 전체 Job이 중단되었습니다. 이를 “건 단위 복구 불가능 구조”로 정의하고, SkipPolicy와 RetryPolicy를 적용해 건 단위 오류 복구 구조를 구현했습니다.

데이터 필수값 검증을 위한 InvalidItemFilterProcessor를 추가하고, 사업자 기준정보 매핑으로 누락·오류 데이터를 자동 보정했습니다.

→ 운영 중단 없는 무중단 지급 배치 구조 확보

[기술] Spring Batch, JPA, Custom SkipPolicy, Logging/Alarm System

<마감 집계 최적화 및 일관성 자동화>

마감 집계는 대량 데이터를 단일 트랜잭션으로 처리하여 장애 시 전체 롤백되는 구조였습니다. 청크 크기와 커밋 간격을 최적화하고, 집계와 상태 업데이트를 분리해 부분 재처리가 가능하도록 했습니다.

→ 집계속도 30% 향상, 누락률 0%, 완전 자동화형 마감 프로세스 구축

[기술] Spring Batch, Oracle 19c, JDBC Template

<가맹점 파일 제공 시스템 리팩토링>

기존 SCP 수동 전송 방식을 REST API 기반 자동화 구조로 전환했습니다. 파일 생성–검증–전송–삭제를 일관된 파이프라인으로 구성하고, Caffeine Cache로 반복 요청 부하를 최소화했습니다.

→ 재전송 대응시간 3분 → 10초, 운영 효율 대폭 향상

[기술] Spring Boot 3, REST API, Spring Batch, Caffeine Cache, Oracle

<테스트 환경 및 멀티모듈 아키텍처 도입>

단일 모듈 구조에서 Core/Domain/Batch로 분리해 책임을 명확히 했습니다. 테스트 전용 H2·PostgreSQL 환경을 구성하고, EntityFixtureGenerator로 복잡한 데이터를 자동 생성했습니다.

→ 빌드시간 25% 단축, 테스트 작성 공수 대폭 감소

[기술] Spring Boot, Gradle Multi-module, H2, PostgreSQL, Spring Batch Test

<모니터링 및 알림 시스템 구축>

배치 오류 추적이 어려웠던 구조를 개선하기 위해 각 StepExecution 이벤트를 수집하여 Telegram 알림으로 통합하고, 정산별 Job 상태와 실패건수를 실시간 모니터링했습니다. Metabase를 도입해 검증 쿼리 70여 개를 시각화했습니다.

→ 장애 대응시간 수십분 → 1분, 데이터 검증 가시성 확보
